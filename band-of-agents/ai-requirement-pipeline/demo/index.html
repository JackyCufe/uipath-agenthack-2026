System Design

Pipeline 设计哲学

为什么这样设计？六个 Agent 各司其职，人始终是最终决策者。

01
源头质量优先
需求在入口处被拦截，而非在研发中途返工。守门Agent是整个系统的第一道防线，也是最重要的一道。
02
人机协同，人是决策者
AI生成草稿、提问、分析；每个关键节点由真实的人通过飞书审批卡片确认。系统不替代判断，只是让判断更有依据。
03
结构化胜过自由发挥
每个Agent的输出是严格定义的Schema JSON，Agent之间通过数据契约通信，而非自然语言传话。可审计、可追溯、可重跑。
04
全链路闭环
从需求提交到客户反馈再到复盘，没有信息孤岛。复盘Agent是唯一能看到完整链路数据的节点，用于发现系统性问题。

评委使用指南

系统已在作者飞书账号上真实运行。评委可通过以下方式验证和查阅。

无需飞书账号可验证逻辑

DEMO_MODE=true python demo.py

DEMO 模式跳过飞书 API，所有输出打印到终端，可验证 6 个 Agent 的完整推理链路。
02 · 真实模式说明
为何无法公开账号
完整运行需要：飞书企业机器人账号 + LiteLLM Proxy（本机 4000 端口）+ Vertex AI Claude Sonnet 配额。

演示视频展示了真实飞书卡片审批和多维表格归档的全流程（DEMO_MODE=false 实际运行）。
03 · Stage 1 守门 Agent（已实现）
一拒一过——守门价值一目了然
❌ 被拒「好烦啊，机器人太不智能了」
→ 四问全空，来源不可溯源，第1轮直接拒绝，要求重新提交
✅ 通过「机器人响应太慢，客户不满意」
→ AI追问3项（客户角色/场景/期望指标）→ 用户一次补全 → 第2轮通过，四问完整
守门规则：四问全空→直接拒绝；缺哪追哪；3轮仍缺→强制拒绝；来源不可溯源→追问说明理由
