## 一、作品定位

**名称**：AI企业提效数字团队

**解决什么问题**

AI机器人公司产品链条从售前→PM→研发→测试→发版全程断裂，根本原因有两个：
1. **摩擦力太大**：每个人知道该怎么做，但做正确的事成本太高，所以流程原则上存在、实际上跑不起来
2. **视野局限**：每个人只看自己那一段，没有人能看见跨环节的系统性问题

**统一设计原则**：拒绝让模糊通过

六个环节，每个Agent做同一件事——在信息传递到下一个人之前，主动挑战模糊描述，强制具体化：
- 守门Agent：需求没有清楚的客户/场景/问题/期望结果 → 追问到清楚为止
- 价值转化Agent：验收标准写"提升用户体验" → 追问到可量化为止
- 场景测试Agent：测试用例没有覆盖真实使用路径 → 补充到覆盖为止

**这套系统的价值**

不是AI替代人判断，而是两件事：
1. 把"应该做但摩擦力太大所以没做"变成"每次自动发生"
2. 把"每个人只看自己那一段"变成"全链路模式对团队可见"

**谁会用**：售前、PM、测试负责人、产品负责人

**触发场景**：售前把客户原话发给飞书机器人，系统自动判断、流转、通知，直至版本复盘闭环

**为什么这个方向**：
- 官方4个示例（CEO内容/品牌营销/客服/产研开发）均未覆盖"产品决策链条"场景
- 来自真实会议纪要（2026-04-27）中明确识别的痛点，非假设场景
- Demo输入来自Jacky实际交付过的真实需求，非虚构场景

## 二、6个Agent架构

### 数据流总览

```
售前把客户原话发给飞书机器人
   ↓
[01 守门Agent] → 真/伪判断
   ├─ 伪需求 → 飞书追问售前 → 补充信息 → 重新判断（最多3轮，否则标记"信息不足，暂不立项"）
   └─ 真需求 → 写入飞书多维表格 → 飞书通知PM
   ↓ （人工确认节点：PM确认接收）
[02 价值转化Agent] → 客户验收标准草稿
   └─ 飞书推送给PM → PM修改确认 → 写入多维表格
   ↓
[03 场景测试Agent] → 客户场景测试用例
   └─ 飞书推送给测试负责人 → 确认后写入多维表格
   ↓
[04 发版评审Agent] → 发版前自检报告（通过/拦截）
   └─ 拦截 → 飞书通知产品负责人，阻断发版；通过 → 飞书发布发版通知
   ↓ （发版后）
[05 反馈收集Agent] → 结构化反馈问卷（自动生成，发给客户）
   ↓
[06 复盘分析Agent] → ROI复盘报告 + 下版本建议
   └─ 飞书推送给产研团队，写入多维表格归档
```

> 飞书多维表格是系统状态中枢：每条需求的当前阶段、负责人、审批记录全部留存，任何人可随时查看整个版本的需求进展。

### 各Agent详情

| # | Agent名称 | 对应痛点 | 输入 | 输出 |
|---|-----------|---------|------|------|
| 01 | **守门Agent** | 售前无法判断真伪需求 | 原始需求描述 | 真/伪判断 + 追问问题清单 |
| 02 | **价值转化Agent** | PM只盯节奏、不转化价值 | 经过验证的需求 | 「客户验收标准」定义 |
| 03 | **场景测试Agent** | 测试按功能点、不模拟客户 | 验收标准 + 产品功能描述 | 客户真实使用场景测试用例 |
| 04 | **发版评审Agent** | 发版无数据支撑 | 所有需求验收情况 | 发版前自检报告（通过/拦截） |
| 05 | **反馈收集Agent** | 发版后无结构化反馈 | 本版本核心交付物 | 结构化反馈问卷 + AI汇总分析 |
| 06 | **复盘分析Agent** | 无闭环、无改进 | 客户反馈数据 + 版本数据 | ROI复盘报告 + 下版本改进建议 |

## 三、Demo Pipeline（完整演示场景）

**背景**：某AI机器人公司，售前把两条真实需求 + 两条伪需求发给飞书机器人，演示完整流转流程。

**输入**（真实需求，Jacky实际交付过）：
```
需求A（真）: 售前需要一个接待后台Skill，给代理商，帮助用户一句话配置好接待后台首页
需求B（真）: 将openclaw与机器人的链路打通，实现用户可以通过飞书下达指令，让机器人完成一些复杂动作
需求C（伪）: 客户说希望机器人更智能（无场景、无具体期望）
需求D（伪）: 代理商反映产品不好用（无场景、无问题定位）
```

**跑一遍展示**：

1. **守门Agent**：需求C/D被拦截，飞书推送追问清单给售前；需求A通过（目标用户/场景/期望结果明确）；需求B被追问——不是因为流程不规范，而是探索性任务本身边界未知，守门Agent将已知部分结构化，将真正未知的部分显式标记，而非让研发自己摸索
2. **价值转化Agent**：为需求A生成验收标准草稿（如「代理商可通过一句话指令完成首页配置，配置成功率≥95%，操作步骤≤3步」），飞书推送PM确认修改
3. **场景测试Agent**：为需求A生成3个场景用例（代理商首次配置、修改已有配置、配置异常回滚），飞书推送测试负责人
4. **发版评审Agent**：逐条检查验收情况，输出通过/拦截报告，飞书通知产品负责人
5. **反馈收集Agent**：展示系统自动生成的结构化客户反馈问卷（机制演示，展示问卷模板与字段设计）
6. **复盘分析Agent**：基于接待后台Skill实际交付情况（Jacky估算数据），生成复盘报告，飞书推送产研团队

## 四、Before vs After 量化对比

| 维度 | 传统方式 | AI数字团队 |
|------|---------|-----------|
| 需求真伪判断 | 售前主观判断，无标准，约30分钟/条 | 3分钟/条，输出结构化判断报告 |
| 验收标准制定 | PM凭经验，1-2天 | 每个需求15分钟，当天完成 |
| 测试用例生成 | 测试工程师手写，2-3天 | 自动生成，30分钟覆盖主场景 |
| 发版前检查 | 依赖个人经验，常有遗漏 | 自动逐条检查，输出拦截报告 |
| 客户反馈收集 | 口头收集，无结构，难分析 | 结构化问卷，AI自动分析汇总 |
| 版本复盘 | 每次2-3小时会议，效率低 | 自动生成复盘报告，会议缩短至30分钟 |
| 需求命中率 | 约40%（历史数据估算） | 目标≥75% |

## 五、技术选型

- **AI模型**：Claude Sonnet 4.6（主力，通过Anthropic SDK调用）
- **人机交互层**：OpenClaw + 飞书机器人（售前/PM/测试负责人通过飞书消息与系统交互）
- **状态中枢**：飞书多维表格（每条需求的阶段、负责人、审批记录全程留存）
- **Agent编排**：Python脚本（负责6个Agent的顺序调用与条件判断）
- **Agent间数据流**：结构化JSON（需求ID + 阶段状态 + 判断结果 + 审批记录）
- **工程亮点**：① JSON Schema强校验确保Agent间数据契约稳定；② 守门Agent设计明确判断rubric（4项必答问题）；③ 每个关键节点设置人工确认点，AI辅助而非替代决策
- **展示页**：静态网页，展示Pipeline流程 + 飞书交互截图 + Before/After对比

> 架构选择说明：本系统6个Agent单向串联，决策点为条件退出而非状态回溯，Pipeline + 条件判断比LangGraph更匹配此场景，且更易在Demo中讲清楚。
