## 需求A — 真需求，守门直接通过

**原始输入文本（售前发给飞书机器人的原话）：**

| 字段 | 提取内容 | Rubric判断 |
|------|---------|-----------|
| `customer_who` | 代理商 | ✅ 通过 |
| `usage_scenario` | 代理商配置接待后台首页时 | ✅ 通过 |
| `problem` | 需要上门服务教客户配置，交付和售后成本高 | ✅ 通过 |

## 需求B — 真需求，守门追问（探索性边界未定 + 内部来源）

**原始输入文本：**
```
将openclaw与机器人的链路打通，实现用户可以通过飞书下达指令，让机器人完成一些复杂动作。目前近景语音交互有问题：机器人收到的消息传不到openclaw。
```

**守门Agent判断过程：**

| 字段 | 提取内容 | Rubric判断 |
|------|---------|-----------|
| `customer_who` | 终端用户 | ✅ 通过 |
| `usage_scenario` | 用户在机器人附近通过语音下达指令时 | ✅ 通过 |
| `problem` | 近景语音交互时机器人收到的消息传不到OpenClaw，链路断掉 | ✅ 通过 |
| `expected_outcome` | "让机器人完成一些复杂动作" — 范围未定义 | ⚠️ 追问 |
| `requirement_source` | 老板/战略（内部新项目） | ⚠️ 额外追问：外部受益场景是什么？ |

**verdict：`info_needed`，round 1**

**追问清单（飞书推送给售前）：**
1. "请描述机器人需要完成的「复杂动作」具体指哪类任务？（如：多步骤导航、搬运序列、展厅讲解流程）"
2. "这个需求的最终受益客户是谁？是为了支撑某个具体客户场景，还是内部能力建设？"

**Demo说明：**
需求B展示守门Agent的高级判断能力——识别出前三个字段已明确，精确定位未明确的边界，并处理内部来源需求的额外验证。区别于C/D的全字段缺失。

## 需求C — 伪需求，守门追问后拒绝

**原始输入文本：**
```
客户说希望机器人更智能
```

**守门Agent判断过程：**

| 字段 | 提取内容 | Rubric判断 |
|------|---------|-----------|
| `customer_who` | "客户" — 模糊泛称 | ⚠️ 追问 |
| `usage_scenario` | 无 | ⚠️ 追问 |
| `problem` | 无具体障碍，只有愿望 | ⚠️ 追问 |
| `expected_outcome` | "更智能" — 纯形容词 | ⚠️ 追问 |

**verdict：`info_needed`，round 1 → 售前无法补充 → round 3 → `rejected`**

**最终 reject_reason：**
```
"信息不足，暂不立项。
缺失信息：具体客户类型、使用场景、遇到的具体障碍、可观测的期望结果。
建议售前重新收集完整需求信息后再提交。"
```
