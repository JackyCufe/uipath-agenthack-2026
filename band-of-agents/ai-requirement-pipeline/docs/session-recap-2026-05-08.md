## 一、关键决策 + 决策理由

### 1. 技术选型：Python Pipeline，不用 LangGraph / CrewAI

**决策**：简单 Python Pipeline + 条件退出

**理由**：
- 6个Agent是单向价值传递，方向不逆转
- "守门拦截伪需求"是条件退出，不是循环；"发版评审拦截"是条件拦截，不是回溯
- 条件退出 ≠ 需要状态图，LangGraph 反而把清晰的流程搞复杂
- Demo里更好讲清楚

**被否定的备选项**：
- LangGraph：适合状态复杂、需要回溯的场景，这个项目不符合
- CrewAI：擅长"分配任务、顺序执行"，但条件分支弱
- AutoGen：已进入维护模式，不推荐

---

### 2. 工程亮点方向

**决策**：三个真实的工程决策作为技术亮点，不堆框架

1. **JSON Schema 强校验 + Structured Output**：Agent 间数据契约，防止链路断裂
2. **守门 Agent 判断 rubric + few-shot examples**：让判断稳定、可解释、可审计
3. **Human-in-the-loop 设计**：关键节点暂停等待人工确认，通过飞书推送

**核心判断**：评委看的是"你意识到了哪些工程问题并解决了它们"，不是用了什么框架

---

### 3. 人机协作界面：飞书机器人

**决策**：OpenClaw + 飞书机器人 = Human-in-the-loop 的交互层

**交互逻辑**：
```
守门Agent跑完
→ 飞书推送给售前："需求信息不完整，请补充：客户场景？期望结果？"
→ 售前在飞书里回复
→ 系统收到回复，继续往下跑
```

**额外设计**：守门最多追问3轮，超过则标记「信息不足，暂不立项」，不无限循环

---

### 4. 多维表格的定位：状态中枢，不是协作工具

**决策**：多维表格是系统状态记录，AI代替PM自动填写，不是所有人手动维护

**基于真实数据的发现**：
- 现有表格771条记录，PRD覆盖率不到50%，大片字段为空
- 场景测试（③）、复盘分析（⑥）：**结构性缺失，连字段都不存在**
- 表格现在是PM专属任务看板，其他人不编辑
- 结论：这不是没填好的问题，是系统设计问题

**权限设计**：产品可编辑，研发+售前只读 → 其他角色通过飞书消息与系统交互，AI代PM写表

---

### 5. 字段设计哲学：从流程需求出发，不从现有字段优化

**决策**：按六个流程环节从头设计字段，用现有字段做命名参考

**理由**：
- 现有表格设计目标是"PM管理自己的工作"，目标错误，不值得继承
- 现有表格基本没跑起来，不是可靠的基线
- 有价值的字段（客户类型、所属场景、需求负责人、版本号）拿过来就好

**两层不可修改机制**：
- `original_text`（售前原话）：永久不可修改，是证据链起点
- 其他字段：可追加、不可删除，每次修改带时间戳和修改人

---

### 6. 作品定位：两层价值

**第一层（执行层）**：把"应该做但摩擦力太大所以没做"的事，变成摩擦力接近零

**第二层（感知层）**：每个角色按结构化模板记录，复盘Agent跨环节合成，让团队看见任何个人都看不见的因果链

**核心叙事**：不是"AI替代人判断"，而是两件事：
1. 把"应该做但没做"变成"每次都自动发生"
2. 把"每个人只看自己那一段"变成"全链路模式可见"

## 二、可复用设计原则（to B 咨询方法论）

### 原则1：定义性判断 vs 执行性判断

- **执行性判断**（有规律可循，可从上下文推导）→ 交给AI
- **定义性判断**（涉及"什么算对"、"什么算够好"）→ 人必须提前锁死

**最常见的设计错误**：把定义性判断也交给AI，导致输出变成通用答案

**在这个项目的体现**：AI生成验收标准的结构和维度，数字标准留空由PM锁定；不能让AI直接给出"配置成功率≥95%"让PM"确认"

---

### 原则2：守门不只是一个环节，是统一设计原则

每个环节的Agent都做同一件事——在信息传递到下一个人之前，主动挑战模糊描述：

- 守门Agent：这个需求描述清楚客户是谁了吗？
- 价值转化Agent："提升用户体验"可以具体化吗？体验的哪个维度？用什么数字衡量？
- 场景测试Agent：测试用例覆盖了客户真实使用路径吗？
- 复盘Agent：不允许输出"总体良好"，必须指出具体哪条需求在哪个环节出了问题

**核心原则：拒绝让模糊通过**

---

### 原则3：系统设计的价值不是拦截，而是让不合规变得可见

- 强行放行是真实的人性问题，系统不可能硬拦
- 正确做法：每次绕过节点的记录进多维表格，复盘Agent自动统计，在复盘报告里直接显示
- 把"偷懒"从隐性变成显性，比实时惩罚更现实、更有说服力

**延伸**：追责机制的设计不是为了惩罚，而是为了让因果链可见

---

### 原则4：to B 项目切入点方法论

1. **切入点**：从一次真实会议纪要提取痛点，不是假设场景
2. **Before量化**：读内部现有工具的数据，用数字说话（样本771条，PRD覆盖率<50%）
3. **目标定位**：区分"自动化"和"让流程真正发生+跨环节模式可见"——后者更深
4. **设计顺序**：Schema → 人机协作节点 → Agent Prompt → 展示层
5. **判断分工**：守门Agent不做判断、做结构化；把信息整理好，让人判断得更快

---

### 原则5：AI与人协作的正确分工

| 由AI做 | 由人做 |
|--------|--------|
| 结构化追问（提问，不判断） | 最终通过/拒绝 |
| 生成验收标准草稿（维度+结构） | 锁定具体数字 |
| 生成测试用例 | 确认可执行性 |
| 生成发版自检报告 | 拍板发版/延期 |
| 生成反馈问卷 | 决定发给谁、什么时机 |
| 跨环节合成复盘报告 | 确认下版本方向 |

**核心原则**：AI降低做正确事情的摩擦力，人保留所有需要判断的决策权

---

### 原则6：原始信息是证据链的起点，不能经过人工过滤再进系统

**核心问题**：现有流程里，所有需求都汇总到PM，PM处理后录入表格 → 原始信息经过一层过滤，失真不可避免

**设计原则**：
- 信息越靠近源头越真实，经过的人越多越衰减
- 售前原话（original_text）必须直接进系统，永久不可修改
- 复盘时可以对比「原始一句话」和「最终交付结果」，看见信息在哪个环节发生了变形
- PM的价值不是录入，是判断——把录入交给AI，把判断留给人

**在这个项目的体现**：original_text 字段不可修改；AI从飞书原始消息提取，不经PM转述

---

### 原则7：现有工具没跑起来，不是工具的问题，是驱动机制的问题

**核心洞察**：多维表格存在，但大片字段为空 → 不是人不知道该填，是填的成本太高、没有人被强制要求填

**在评估「要不要引入AI」之前，先问「现有工具为什么没跑起来」**：
- 如果是「没有驱动机制」→ 加驱动机制，不换工具
- 如果是「工具根本不适合这个场景」→ 才考虑换

**对to B项目的启示**：
- 客户说「我们已经有XX了」≠「XX在用」
- 先读数据、量化Before状态，才能判断是工具问题还是执行问题
- 这也是销售切入点：「你们有这个工具，但它跑起来了吗？」

---

### 原则8：AI系统的交付边界要在提案中清晰分层

**设计原则**：作品/提案里必须说清楚「已实现」vs「设计蓝图」，不能含糊

- 已实现：Agent逻辑能跑起来，输出结果可展示
- 设计蓝图：机制正确、但需要真实数据才能运行的部分（如反馈汇总、跨版本复盘）

**原因**：
- 评委/客户能接受「这部分是蓝图，需要3个月真实运营数据才能跑」
- 评委/客户无法接受「说能做但其实不能」——被拆穿的代价比诚实大得多
- 清晰的边界本身就是专业度的体现

---

### 原则9：探索性任务 ≠ 伪需求，两者处理方式不同

**核心区分**：

| | 伪需求 | 探索性任务 |
|---|--------|-----------|
| 本质 | 信息不完整 | 目标本身边界未知 |
| 例子 | "客户说希望机器人更智能"（无场景无期望） | "龙虾餐厅机器人"（场景清楚，但交互边界未定义） |
| 正确处理 | 追问到四要素齐全 | 结构化已知部分，显式标记未知部分 |
| 错误处理 | 放行（信息不够就过） | 拦截（这不是伪需求） |

**设计原则**：守门Agent对探索性任务的价值是「把已知部分结构化，把真正未知的部分显式标记」→ 让研发在已知框架内探索，而不是盲目摸索

---

### 原则10：系统的差异化价值在复盘Agent，不在守门Agent

**容易犯的错误**：把太多注意力放在第一个环节（守门），忽略后面5个

**真正的差异化**：
- 守门 → 任何有纪律的人工流程都能做
- 复盘Agent跨6个环节合成 → 这是人工永远做不到的，因为没有人能同时看到全链路

**类比**：self-exploration里的核心不是「收集数据」，是「AI发现用户自己看不见的模式」；这里的核心不是「追问需求」，是「复盘Agent发现团队自己看不见的因果链」

**实际案例**：守门时某需求场景信息缺失 → 测试时这个需求用例覆盖不足 → 发版后客户反馈恰好是场景匹配差，这三个点分散在不同环节，没有人能同时看到，复盘Agent把它们连起来

## 三、当前系统设计状态（快照）

### 6个Agent架构

| # | Agent | 统一原则的体现 | 人工确认点 |
|---|-------|-------------|-----------|
| 01 | 守门Agent | 需求4要素缺任何一个→追问（最多3轮） | 售前确认通过/不立项 |
| 02 | 价值转化Agent | "提升体验"→追问到可量化维度 | PM锁定验收标准数字 |
| 03 | 场景测试Agent | 测试用例必须是真实使用路径 | 测试负责人确认可执行 |
| 04 | 发版评审Agent | 无验收标准+测试结果→自动拦截 | 产品负责人决策发版/延期 |
| 05 | 反馈收集Agent | 问卷必须对应具体验收标准，不能问"满意吗" | 售前确认发送对象和时机 |
| 06 | 复盘分析Agent | 不输出"总体良好"→指出具体因果链 | 产研团队确认下版本方向 |

### Demo输入数据

```
需求A（真）：售前需要一个接待后台Skill，用于展厅场景下接待访客
需求B（真，探索性）：龙虾餐厅场景的机器人点餐与引导Skill
需求C（伪）：客户说希望机器人更智能（无场景、无具体期望）
需求D（伪）：代理商反映产品不好用（无场景、无问题定位）
```

### 5个JSON Schema（已起草）

- Schema 1：守门Agent → 价值转化Agent（含 original_text 不可修改）
- Schema 2：价值转化Agent → 场景测试Agent（含 value_type / importance，数字由PM填）
- Schema 3：场景测试Agent → 发版评审Agent（含完整 test_cases 结构）
- Schema 4：发版评审Agent → 反馈收集Agent（含 bypass_log 告警字段）
- Schema 5：反馈收集Agent → 复盘分析Agent（含 criteria_covered / met_criteria）

### 技术栈

- 编排层：Python Pipeline + 条件退出
- 执行层：OpenClaw
- 人机交互：飞书机器人（追问 / 确认通知 / 拦截告警）
- 状态中枢：飞书多维表格（AI代写，PM确认）
- 模型：Claude claude-sonnet-4-6

## 四、Jacky 的认知跃迁时刻

1. **守门不是第一个Agent的专属，是所有Agent的统一原则**
   → 每个环节都在做同一件事：拒绝让模糊通过

2. **飞书机器人是现成的Human-in-the-loop界面**
   → 之前没想到，意识到后立刻打通了整个系统设计

3. **多维表格是状态中枢，不是协作工具**
   → 区别不是表格本身，而是谁来驱动流程；AI驱动 vs 人想起来才填

4. **系统真正的价值不是自动化，是让全链路模式可见**
   → 对应 self-exploration 里「结构化收集 → 让用户被看见」的核心机制
   → 这里是「结构化记录 → 让团队看见自己看不见的因果链」

5. **这是to B AI实施落地的第一个案例**
   → 基于真实内部数据验证，严格意义上是内部项目，但方法论成立
   → 可以说"曾为某AI机器人公司设计需求管理AI Pipeline，基于真实内部数据验证"

6. **探索性任务和伪需求是两种不同的东西**
   → 需求B（龙虾机器人）不是信息不完整，是边界未定义，处理方式完全不同

7. **系统的叙事背后有真实的血泪——离职的同事**
   → 一句话需求进来→没头绪→压力过大→离职，这不是个人问题，是系统问题
   → 这是「如果这套系统当时存在会怎样」最具体、最有力的回答

## 五、Jacky 的个人语言表达风格

> 用于以后写作/内容创作时作为风格参考

### 提问方式
- 喜欢用「等会」打断，然后提出一个更底层的问题
- 经常问「这有啥区别？」「这是不是自嗨？」「这会不会太简单了？」——追问本质
- 用「嘶」表示正在思考、有新的关联浮现
- 不接受表面答案，会追问「你说的到底能不能做到？」

### 思维方式
- 边做边质疑，不会完全接受 AI 的答案
- 重视「做这件事的理由」，不只要结论，要问为什么
- 会主动提出反例来验证设计（「如果人强行放行呢？」）
- 担心「越权」「自嗨」「太简单」——有清醒的自我认知
- 喜欢用已有概念做类比（把这个项目类比到 self-exploration 的「被看见」机制）

### 表达习惯
- 口语化，句子短，用逗号断句多
- 直接叫停：「等会啊」「先别做」「不不不」
- 给 AI 留余地：「你客观分析一下」「别迎合我」「你先说说你的逻辑」
- 复述确认：「我理解得对吗？」「我复述一遍：...」
- 开心时会直接说：「内心有点开心」
- 喜欢问「这有没有漏掉什么？」——不放过细节

### 典型句式
- 「这玩意跟XX有什么区别呢？」（挑战系统存在的必要性）
- 「我有点不懂了」（重新回到起点，重新建立理解）
- 「诶等下，我突然想到」（新关联突然浮现）
- 「好，那下一步呢？」（目标导向，不停在讨论里）
- 「你先说思路，我来决定」（明确分工）
- 「我觉得这是一个真需求」（自己有判断，不等 AI 背书）

### 做事风格
- 喜欢「先讨论清楚再动手」，但会在讨论偏长时自我纠正「好，开干」
- 会主动做侦察（问PM、看多维表格），然后把信息带回来分析
- 不满足于"方向对了就行"，要把每个细节都想清楚
- 对自己的作品有责任感：「我们真的能做到 After 写的这样吗？」
