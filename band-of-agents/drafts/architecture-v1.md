# 架构图 V1 — 基于Band的客户反馈路由系统

## 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                         用户侧                               │
│                                                              │
│   客户在Lark群发送后续反馈                                      │
│   "上次那个搜索功能最近又出问题了"                                │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                     Lark Bridge（桥接层）                      │
│                                                              │
│   收Lark消息 → 转发到Band Room → @routing-agent                │
│   单向转发，不做任何判断                                         │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                      Band Room                               │
│                                                              │
│   ┌─────────────────────────────────────────┐               │
│   │         @routing-agent                  │               │
│   │                                         │               │
│   │  1. 查Bitable历史：这条反馈对应哪条需求？  │               │
│   │  2. 诊断问题：技术bug / 运营服务 /         │               │
│   │              售前新需求 / 售后投诉？       │               │
│   │  3. 判断切入点：从哪个Stage开始处理？       │               │
│   │  4. @mention 对应的Pipeline Agent       │               │
│   └─────────────┬───────────────────────────┘               │
│                 │                                            │
│                 │  @mention 对应Agent                         │
│                 ▼                                            │
└─────────────────────────────────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────────────┐
│                  AI需求链（Lark飞书侧，不动）                     │
│                                                              │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│   │ S1守门   │  │ S2价值转化│  │ S3场景测试│                 │
│   └──────────┘  └──────────┘  └──────────┘                 │
│                                                              │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│   │ S4发版评审│  │ S5反馈收集│  │ S6复盘分析│                 │
│   └──────────┘  └──────────┘  └──────────┘                 │
│                                                              │
│   被@mention的Agent从指定Stage启动                             │
│   → 发Lark卡片给对应负责人                                     │
│   → 人审批/处理                                               │
│   → 结果写入Bitable                                           │
└─────────────────────────────────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────────────┐
│                  飞书 Bitable（数据层）                         │
│                                                              │
│   ┌─────────────────────────────────────────┐               │
│   │  每条需求从S1到S6的完整档案                │               │
│   │  - 原始需求描述                           │               │
│   │  - 守门结论                               │               │
│   │  - 验收标准                               │               │
│   │  - 测试结果                               │               │
│   │  - 发版记录                               │               │
│   │  - 客户反馈                               │               │
│   │  - 复盘报告                               │               │
│   └─────────────────────────────────────────┘               │
│                                                              │
│   routing-agent 读取历史上下文                                │
│   Pipeline Agent 写入处理结果                                 │
└─────────────────────────────────────────────────────────────┘
```

## 数据流时序

```
客户 Lark消息
    │
    ▼
Lark Bridge
    │ 转发，不判断
    ▼
@routing-agent（Band Room）
    │
    ├──→ 查Bitable历史 ──→ "这是REQ-089的后续反馈"
    │
    ├──→ 诊断问题性质 ──→ "技术类bug，非新需求"
    │
    ├──→ 判断切入点 ──→ "从S3场景测试切入"
    │
    └──→ @S3-agent ──→ "REQ-089回归bug，接验收标准跑测试用例"
                          │
                          ▼
                    S3-agent发Lark卡片给测试负责人
                          │
                          ▼
                    测试负责人审批/执行
                          │
                          ▼
                    结果写入Bitable
```

## Band 核心价值体现

| Band优势 | 体现 |
|---------|------|
| 跨框架Agent协作 | routing-agent（Band）调度Pipeline Agent（Lark） |
| Agent动态发现 | routing-agent在Band目录里发现可用Pipeline Agent |
| @mention路由 | routing-agent根据问题内容决定@谁，不是硬编码if/else |
| 上下文共享 | routing-agent把历史上下文完整传给目标Agent |
| 人在例外处 | Agent自己判断路径，只有真正需要处理时才发卡片给人 |
